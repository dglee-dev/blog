# PostList Skeleton 설계

## 목적

`/posts` 진입 시 데이터를 fetch하는 동안 빈 화면 대신, 실제 리스트와 동일한 셰이프의 스켈레톤을 보여준다. prefetch 캐시가 있으면 즉시 렌더링되지만, 첫 방문이거나 prefetch가 완료되기 전이라면 스켈레톤이 필요하다.

스켈레톤은 실제 포스트 개수와 동일한 개수로 표시되어야 의미가 있다. 고정된 개수의 플레이스홀더는 실제 렌더링 시 레이아웃이 튀기 때문이다.

---

## 스켈레톤 데이터 타이밍

실제 포스트 목록의 셰이프를 맞추려면 포스트 개수를 미리 알아야 한다. 이 데이터를 언제, 어디서 가져올지가 핵심 문제다.

**빌드타임 생성**

빌드 시 S3에서 목록을 가져와 JSON으로 번들에 포함하는 방식. 첫 방문에도 사용 가능하지만, 글이 추가되었을 때 빌드+배포를 같이 해야 셰이프가 맞는다. 글 추가와 배포가 항상 함께 이루어진다는 보장이 없어서 틀어질 가능성이 있다.

**localStorage**

한 번 fetch한 결과를 저장해두고 다음 방문 시 사용하는 방식. 첫 방문에는 사용할 수 없다.

**서버 캐시 (Vercel KV)**

Lambda가 S3에서 포스트 목록을 가져올 때 Vercel KV에 캐시해두고, 클라이언트는 KV에서 먼저 읽는 방식. 첫 방문에도 사용 가능하고, 글이 추가되면 다음 요청 시 자동으로 반영된다.

---

## 선택: Vercel KV

첫 방문 문제를 해결하면서 글 추가 시 자동으로 반영되는 유일한 방식이다. 빌드타임은 배포 의존성이 생기고, localStorage는 첫 방문을 커버하지 못한다.

Vercel KV는 Upstash Redis 기반의 관리형 서비스로, 지금 프로젝트가 Vercel에 배포 중이라 통합이 자연스럽다.

---

## 구현 방향

```
사용자 접속
  → Lambda: Vercel KV에서 캐시된 목록 반환
  → 클라이언트: 캐시 목록으로 스켈레톤 즉시 표시
  → 동시에 Lambda: S3에서 최신 목록 fetch
  → length 비교 → 다르면 KV 업데이트
  → 실제 데이터 도착 → 스켈레톤 교체
```

**Lambda 변경**
- 요청 시 KV에서 캐시 목록 읽어서 즉시 응답
- 백그라운드에서 S3 목록 fetch → 캐시와 length 비교 → 다르면 KV 업데이트

**클라이언트 변경**
- Lambda 응답에 캐시 목록 포함 시 스켈레톤 즉시 렌더링
- 실제 데이터 도착 시 교체

**캐시 업데이트 기준**

length 비교로 충분하다. 글이 삭제되는 경우는 거의 없고, 추가가 주된 변경이기 때문이다.
