## 들어가며

자바스크립트는 복잡하다. 변화에 무한대로 열려있는 웹이라는 환경에서 실행되어야하고, 누군가 한 두명이 만들어서 배포하는게 아니라 마치 민주주의에서 법을 정하듯 위원회에서 언어의 규칙을 결정한다. 동적 언어로 설계되어 정적 분석이 어렵다는 등의 초기 설계에서의 태생적 한계도 존재한다.

따라서 자바스크립트는 1995년 탄생 이후 많은 변화를 겪어왔다. 비동기 요청의 탄생과 발전, V8, ES6, CommonJS, npm, TypeScript까지.. 여러 방면으로 확장하고 합의되며 그 영향력을 키워가고 있다. 그러므로 자바스크립트의 이 모든 변화를 하나의 주제로 이해하기란 어렵다.

하지만 가장 실용적이고 핵심적인 변화를 꼽자면 바로 자바스크립트의 모듈시스템이다. 본디 가벼운 스크립트 언어에서부터 '진지한 프로그래밍 언어' 로 발전하는 과정의 중심에는 모듈시스템이 있으며, 현대에도 이 모듈시스템을 둘러싼 발전은 진행형이다. 매일의 코드베이스에서도 우리는 아주 쉽게 이 발전의 흔적과 현재 진행중인 고민들을 확인할 수 있다.

개인적으로도 해당 주제에 닿고 본격적인 정리를 해야할 필요성을 느끼게 된 이유 또한 '매일 마주하는데 완벽히 이해하지 못하는' 상태를 아주 오랫동안 겪어왔기 때문이다. CJS와 ESM의 문법이 무엇이고 어떻게 사용되는지를 100번 찾아도 101번째 다시 찾게 되는 이유는 둘의 본질적인 차이와 그 맥락에 대한 몰이해 때문이라고 판단했다. tsconfig의 esModuleInterop 옵션을 설정할때도 이 설정이 왜 필요한 것인지 완전히 이해하지 못한 채 그저 필요하니까 켜고 끄는 일을 반복하며 죄책감을 느껴왔다.

해당 글은 시리즈로 작성될 예정이다.

1. 먼저 모듈시스템의 발전을 중심으로 자바스크립트의 발전 역사에 대해 알아보고,
2. 현행으로 사용되고 있는 모듈 시스템인 CJS와 ESM의 사용 방법과 설계의 특징에 대해 알아본다.
3. 마지막으로 이들의 Interoperability라는 현재 진행중인 문제를 ts나 번들러에서 어떻게 다뤄내고 있는지, 우리는 이 상황에서 어떻게 모듈을 작성하고 공개하며 조직해야 하는지에 대해 알아본다.

시리즈를 마쳤을 때, 쓰는 사람으로서는 정확한 어휘로 모듈시스템을 설명할 수 있는 사람이 되기를 바란다. 또한 모듈 시스템의 발전을 둘러싼 논의들을 따라가며 기술이 어떤 맥락과 동기를 기반하여 앞으로 나아가는지에 대해 이해하는, 기술안에 있는 사람들의 흔적을 알고 또한 자신의 결정에 그러한 기준들을 활용할 수 있는 사람이 되기를 바란다.

읽는 이는 왜 되거나, 왜 안되는지에 대한 이유를 알 수 있기를 바란다. require와 import는 왜 같이 사용할 수 없는지, package.json의 "type": "module" 이 어떤 상황에서 필요하며 정확히 무엇을 바꾸는지, esModuleInterop이 왜 필요한 옵션인지, 등등. 또한 파편화 된 지식들을 하나로 연결할 수 있는 접착제 역할을 할 수 있는 글이 되기를 바란다. CJS, ESM, 번들러, 트랜스파일러 등을 하나의 맥락 안에서 연결짓는다면 이들의 역할과 한계를 보다 확실하게 인식하여 현실세계의 프론트엔드 코드 인프라를 더 자신감있게 다룰 수 있게 되지 않을까.

## 1. AJAX로 인한 웹 애플리케이션의 등장

자바스크립트 코드베이스가 커지게 된 계기는 AJAX 등장 덕분이었다. AJAX는 비동기적으로 데이터를 페칭하고, 가져온 데이터를 바탕으로 화면을 refresh없이 업데이트하는 메커니즘을 의미하는데, 2000년경 이미 Microsoft의 Outlook Web Access에서 이를 이용해 웹에서 데스크탑 앱과 같은 UX를 구현하였지만 대중적으로 AJAX를 이용한 비동기적 화면 업데이트 방식의 웹 '애플리케이션' 을 알린 것은 2004년에 등장한 Gmail이 대표적이다.

이렇게 하나의 웹페이지를 마치 애플리케이션처럼 개발할 수 있게 되면서 UI 상태관리, DOM 조작, 비동기 흐름 제어, 클라이언트 사이드 라우팅 등을 위한 코드가 추가되기 시작했다. 우리가 알고 있는 '프론트엔드 개발' 시대가 시작된 것이다. 이렇게 클라이언트 측 자바스크립트의 코드량이 증가하게 되면서 자연스럽게 모듈 시스템에 대한 필요성이 대두되게 되었다.

## 2. 초기 클라이언트 모듈 시스템 (Yui Loader, Closure Library, Dojo)

대기업들은 그들의 자체적인 기술력을 이용해 클라이언트 모듈 시스템을 제작해냈다. Gmail을 개발한 구글은 이미 2004년경 현대의 Webpack과 같은 역할을 하는 번들링/컴파일 툴인 `Closure Library`를 만들어 사용하고 있었으며, 야후는 [`YUI`](https://clarle.github.io/yui3/) 라는 이름의 라이브러리에 모듈 시스템을 얹어내었다.

하지만 2009년 AMD의 구현체인 Require.js가 등장하기 전까지 일반적인 개발자들은 IIFE와 같은 네임스페이스 구현 수준의 임시방편을 사용하는 것이 전부였다. [`Require.js History`](https://requirejs.org/docs/history.html?utm_source=chatgpt.com)에서 Require.js가 등장하기 전, 개발자들이 어떻게 모듈 관리를 했어야 하는지에 대한 처참한(?) 상황을 엿볼 수 있다.

- Douglas Crockford가 야후 재직 당시 YUI 팀에게 자문하는 아키텍트 역할로서 YUI 2의 Module Pattern을 통한 네임스페이스 구현에 큰 영향을 주었다.

- 그는 Require.js 의 '동적 모듈 로딩' 과 같은 아이디어를 탐탁찮아했다.

- > A module is a function or object that presents an interface but that hides its state and implementation (Douglas Crockford)

## 3. 서버사이드 모듈 시스템(CJS)과 Node.js

- 2009년 Kevin Dangoor의 [`What server side JavaScript Needs`](https://www.blueskyonmars.com/2009/01/29/what-server-side-javascript-needs/)

Node.js가 CommonJS를 채택했다. 이 과정에서 기존 명세에 있던 exports 말고 module.exports 도 추가되었다. 이것을 기반으로 Node.js/NPM 생태계가 만들어지며 CJS 모듈 기반 라이브러리 생태계가 활성화.

## 4. 분리된 클라이언트 모듈 시스템 (AMD, Require.js)

CJS는 동기적으로 모듈을 로드하는 시스템이라서 브라우저에서는 채택이 어려웠다. 하지만 이미 대기업에는 클라이언트에서 사용되는 비동기 모듈 로드 시스템이 구현되어 있었고, 이런 구현들을 표준화 해 브라우저 측에서도 사용할 수 있는 표준 모듈 시스템을 구현하고자 하는 움직임이 일어났다. 그것이 AMD(Asyncronous Module Definition)이다.

Require.js는 AMD의 대표 구현체이다.

## 5. 모듈 번들러(Browserify, Webpack)의 등장

AMD는 2011~2013년 즈음 득세하는 듯 했지만 Browserify, Webpack처럼 'CJS를 번들링해서 클라이언트에 던져주면, 클라이언트에서는 모듈 로딩이 필요없다' 라는 해결책이 나오면서 힘을 잃었다.

Webpack은 CJS 뿐 아니라 AMD 코드도 지원했고, JS뿐 아니라 CSS, 이미지, 폰트 등 정적 에셋까지 모두 모듈로 묶었다(이게 뭔소릴까?).
