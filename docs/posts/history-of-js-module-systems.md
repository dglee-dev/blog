## 들어가며

자바스크립트는 마치 오래된 도시와도 같다. 처음에는 우물가의 작은 마을로 만들어졌지만, 시대의 흐름에 따라 주변 도시의 역할을 흡수하기도 하고 도시의 인프라를 늘려가고 여러 건물을 짓고 허물며 점차 큰 도시가 되었음에도 그 역사적인 흔적들을 간직하고 있는, 깔끔하지는 않지만 높고 낮은 스카이라인을 보이는 세계에서 거의 가장 많은 사람들을 품고 있는 사랑스러운 도시.

자바스크립트가 생겨나서 지금의 모습으로 발전하기까지 AJAX, V8, ES6, Nodejs, TypeScript 등 많은 변화의 순간들이 존재했지만 이 모든 변화의 역사를 함께하며 모습을 바꿔온 인프라가 있다면 바로 **자바스크립트의 모듈시스템**이다. 본디의 가벼운 스크립트 언어에서부터 진지한 프로그래밍 언어로 발전하는 과정의 중심에는 모듈시스템의 등장과 발전의 역사가 존재하며, 2026년 현재에도 이 모듈시스템을 둘러싼 발전은—거의 완결되어가고 있으나—현재 진행형이다. 우리가 작성하는 매일의 코드베이스에서도 우리는 그 발전의 흔적과 현재 진행중인 고민들을 확인할 수 있는데, 대표적으로 TypeScript의 `esModuleInterop`, `allowSyntheticDefaultImports`, `target` 과 같은 옵션이 그것이다. 마치 땅 속에 쌓여있는 지층이 단면을 드러내듯 과거의 흔적이 남아있는 것이다.

모듈시스템의 역사에 대해 충분히 알아보겠다는 마음을 가지고 본격적인 정리의 필요성을 느끼게 된 이유 또한 이러한 지층, 역사적인 흔적으로 생겨난 설정과 개념들을 마주하며 '매일 마주하는데 항상 찝찝한' 상태를 아주 오랫동안 겪어왔고, 그 이유가 과정에 대한 이해 없이 단순히 사용만 하기 때문이라는 생각이 들었기 때문이다. CJS와 ESM의 문법이 무엇이고 어떻게 사용되는지를 100번 찾아도 101번째 다시 찾게 되는 이유는 둘의 본질적인 차이와 그 맥락에 대한 몰이해 때문일 것이라는 생각이 들었다. 이 설정이 왜 필요한 것인지, 설정을 바꾸었을 때 어떤 것이 바뀌는 것이고 그것이 왜 필요한 일인지 정확히 이해하지 못한 채 그저 에이전트가 시키는대로 켜고 끄는 일을 반복하며 죄책감을 느껴왔다.

나는 AI 시대에도 여전히 개발자가 필요하다고 믿고, 개발자의 역할은 결국 그가 작성한 코드를 믿고 맡길 수 있는 사람인 것이라고 생각한다. AI에게 믿고 맡길 수 있을 정도로 작고 일상적이고 큰 손해가 발생하지 않는 일들은 AI에게 맡기면 되겠지만 비즈니스를 구성하며 의도가 명확해야하는 `인간을 위한 코드` 는 인간이 책임져야 한다. 그러려면 우리는 이 코드를 작성함으로서 컴퓨터에게는 무슨 일이 일어나고 있는지, 그리고 컴퓨터가 그렇게 하도록 만든 인간들에게는 어떤 의도와 변화와 과정이 있었는지 이해해보는 것이 도움이 되리라. 그것이 이 글을 작성하는 이유이다.

## 구성

해당 글은 3편의 시리즈로 작성될 예정이다.

1. 먼저 **모듈시스템의 발전**을 중심으로 자바스크립트의 역사에 대해 알아보고,
2. 현행으로 사용되고 있는 모듈 시스템인 **CJS와 ESM의 사용 방법과 설계의 특징**에 대해 알아본다.
3. 마지막으로 공존하는 현행 모듈 시스템 간의 **Interoperability**라는 현재 진행중인 문제를 TypeScript 나 번들러에서 어떻게 다뤄내고 있는지, 우리는 이 상황에서 어떻게 모듈을 작성하고 공개하며 조직해야 하는지에 대해 알아본다.

## 목표

3편까지 작성을 마쳤을 때, 쓰는 사람으로서는 정확한 어휘로 모듈시스템을 설명할 수 있는 사람이 되기를 바란다. 또한 모듈 시스템의 발전을 둘러싼 논의들을 따라가며 기술이 어떤 맥락과 동기를 기반하여 앞으로 나아가는지에 대해 이해하는, 기술안에 있는 사람들의 흔적을 알고 또한 자신의 결정에 그러한 기준들을 활용할 수 있는 사람이 되기를 바란다.

읽는 이는 왜 되거나, 왜 안되는지에 대한 이유를 알 수 있기를 바란다. require와 import는 왜 같이 사용할 수 없는지, package.json의 "type": "module" 이 어떤 상황에서 필요하며 정확히 무엇을 바꾸는지, esModuleInterop이 왜 필요한 옵션인지, 등등. 또한 파편화 된 지식들을 하나로 연결할 수 있는 접착제 역할을 할 수 있는 글이 되기를 바란다. CJS, ESM, 번들러, 트랜스파일러 등을 하나의 맥락 안에서 연결짓는다면 이들의 역할과 한계를 보다 확실하게 인식하여 현실세계의 프론트엔드 코드 인프라를 더 자신감있게 다룰 수 있게 되지 않을까.

# **자바스크립트 모듈 시스템의 역사**

### 자바스크립트의 등장과 AJAX

자바스크립트는 Netscape의 브라우저인 Navigator에 탑재되기 위해 개발되었다. 목적은 프로그래머가 아닌 사람들을 위해 동적인 웹사이트를 만들게끔 돕는 것이었다. Netscape에 고용된 브랜든 아이크가 개발하여 채택되었고, 1995년 9월에 LiveScript 라는 이름으로 먼저 릴리즈 되었지만, 마케팅을 목적으로 JavaScript로 그 해 12월 이름이 변경되었다. 초창기의 자바스크립트는 그 기획 의도에 맞게 소소한 위젯이나 간단한 유효성 검사(폼 필드 확인) 용도로 사용되었다.

`모듈` 이라는 기능이 필요할정도로 자바스크립트로 작성된 프로젝트 코드베이스의 규모가 커지게 된 것은 AJAX라는 개념이 대중화되면서부터 라고 볼 수 있다. AJAX는 현대에는 아주 흔한, 비동기적으로 데이터를 페칭하고, 가져온 데이터를 바탕으로 웹페이지를 refresh없이 업데이트하는 메커니즘을 의미하는 단어이다. Asynchronous JavaScript and XML의 약자로서 비동기적으로 XML 데이터를 가져와 화면을 업데이트했던 초기 방식을 담고있다.

이미 데스크탑 애플리케이션의 유저 경험에 익숙했던 개발자와 사용자들은 웹페이지에서 하나의 액션을 취할때마다 페이지를 새로고침해야 한다는 지점에 대해 아쉬움이 많았다. 따라서 페이지를 완전히 새롭게 그리지 않고도 화면을 업데이트하고 싶다는 니즈는 자바스크립트가 생겨나기 이전부터 존재했다. 다르게 말하자면 클라이언트 측 상태를 유지시키고 싶다는 생각을 계속해서 해왔던 것이다.

이에 대한 초기의 시도는 일명 Remote Scripting이라는 방식이었다. iframe으로 현재 보여주는 페이지와 분리된 컨텍스트에서 추가적인 요청과 응답을 처리했다. 이후 1999년 IE5에 탑재되었던 XMLHttp를 거쳐 XMLHttpRequest이 표준화되었고, 2004년 Gmail과 2005년 Google Maps가 등장하며 마치 데스크탑 애플리케이션과 같은 경험을 제공하는 '웹 애플리케이션' 이라는 개념을 대중화시켰다. 2005년에는 Jesse James Garrett에 의해 AJAX라는 단어가 만들어지며 비동기 웹 기술과 유저 경험은 완전히 주류가 되었다.

그렇게 하나의 도메인 주소가 가리키는 것이 '페이지' 가 아니라 '애플리케이션' 에 가까운 것으로 변화하게 되면서 수 많은 새로운 로직들이 필요하게 되었다. 새로고침되지 않아 보존되는 클라이언트 상태를 관리해야 했고, 새로운 데이터를 화면에 반영하기 위한 DOM 조작, 데이터를 보내거나 받아오기 위한 비동기 흐름 제어, 클라이언트 사이드 라우팅 등을 위한 코드가 자바스크립트 코드베이스에 추가되기 시작한 것이다. 그 결과 클라이언트 측 자바스크립트의 코드량이 폭발적으로 증가하고 여러 사람이 파일단위의 코드를 작성하고 임포트하게 되면서 의존성에 대한 명시적인 관리가 필요해졌고, 자연스럽게 모듈 시스템도 대중화를 이루게 되었다.

### 모듈 시스템 이전과 이후

모듈 시스템의 등장에 대해 다루기 전, 모듈과 모듈 시스템을 구분지어보려고 한다. '모듈' 은 파일단위, 혹은 사용자 임의의 네임스페이스를 규정하여 코드끼리의 구획을 나누어 응집시키고 상호작용 시킬 수 있는 단위로 분리하는 방법을 의미한다. IIFE를 이용한 네임스페이스 구분이 대표적이다. '모듈 시스템' 은, 위에서 가볍게 언급한 바와 같이 '의존성에 대한 명시적인 관리' 가 핵심이다. 이미 파일단위의 모듈로 나누어진 코드 사이에 거미줄처럼 얽힌 관계를 사용자가 메타데이터로 표현하고 그것을 기반으로 로드 타이밍 등을 조정하는 기능을 포함한 일명 '모듈 로더' 를 의미하는 것이다. 코드를 나누는 것과, 나뉜 것들을 관리하는 시스템의 차이이다.

### 네임스페이스

언어 레벨에서 모듈의 개념이 구현되어있지 않았던 자바스크립트에서 모듈의 기본적인 역할인 네임스페이스를 구현하는 방법은 크게 객체 네임스페이스를 이용하거나 IIFE를 이용해 독립된 함수 스코프에 네임스페이스를 만드는 방법이 있었다.

```ts
// Object Namespace

const MyModule = {};

MyModule.name = 'module';
MyModule.doSomething = function () { ... };
```

```ts
// Module Pattern

const MyModule = (function () {
  let count = 10;

  const get = function () {
    return count;
  };

  const increment = function () {
    count++;
  };

  return {
    get,
    increment,
  };
})();
```

두 가지 방식은 배타적이지 않고 오히려 함께 사용되는 경우가 많았다. 객체를 기반으로 한 네임스페이스 구분은 충분히 효과적이었으므로 더 자주 사용되었을 것으로 예상되고, 모듈패턴은 구현과 유지보수 및 디버깅이 그보다는 다소 복잡한 면이 있는 방면 확실하게 프라이빗한 데이터를 캡슐화하거나 내가 쓴 코드가 남의 코드베이스에 들어가야 하는 경우 - 예를 들어 라이브러리 - 에 일종의 매너처럼 관습적으로 사용되었을 것으로 예상된다.

### 의존성 문제

그러나 이렇게 단순히 네임스페이스를 구분하는 것 만으로는 불충분한 유즈케이스가 존재했다. 그것은 바로 모듈 간 **의존성**이 복잡해지는 순간이다.
예를 들어, 지메일과 같은 메일 클라이언트 웹앱에서 사용되는 복잡한 로직들을 각각의 모듈로 만든다고 하면 아래와 같이 분리될 수 있을 것이다.

```
core/
api/
mail/
draft/
search/
router/
ui/
keyboard/
cache/
notification/
```

여기에서 의존성을 고려해서 스크립트를 로드하려면 어떤 모듈이 어디에 의존하는지를 기록하는 문서를 작성하게 될 것이다. 그리고 해당 문서에 의거해서 스크립트 순서를 개발자가 수동으로 관리해야한다. 이 부분에서 개발자라면 '작성된 의존성대로 스크립트 순서를 자동으로 계산해주는 알고리즘을 짤 수 있지 않을까?' 라는 생각이 들기 마련이다. 이것이 당시의 `Loader` 의 역할이다. 여기에 중복 로딩을 방지하고, race condition, 로딩 실패처리, 캐시, 실행 순서, circular dependency등에 대한 처리를 추가하다보면 사실상 작은 런타임 시스템이 필요해지는 것이다.

그렇게 대규모 웹 애플리케이션 작성을 위한 초기 클라이언트 모듈 시스템들이 생겨나게 되었다.

## 3. 초기 클라이언트 모듈 시스템

### Google Closure Library, Yahoo User Interface Library

웹페이지의 애플리케이션화로 인해 코드가 복잡해지고 이에 따라 모듈 로더에 대한 필요성이 생겨나면서, 대기업들은 그들의 자체적인 기술력을 이용해 클라이언트 모듈 시스템을 제작해냈다. Gmail을 개발한 구글은 이미 2004년경 현대의 Webpack과 같은 역할을 하는 번들링/컴파일 툴인 `Closure Library`를 만들어 사용하고 있었으며, 야후는 [`YUI`](https://clarle.github.io/yui3/) 라는 이름의 라이브러리에 모듈 로더를 얹어내었다.

### Dojo Toolkit

Dojo는 2004년에 등장한 프레임워크로서, Dijit이라는 이름의 UI 위젯 라이브러리와 Java/C# 친화적인 클래스 기반 OOP 패러다임(dojo.declare), 대규모 자바스크립트 코드베이스를 관리하기 위한 모듈 시스템(sync XHR - AMD), 그리고 i18n과 크로스 브라우징까지 하나로 묶어 제공하는 '엔터프라이즈급' 웹 애플리케이션 제작용 툴킷이었다. Alex Russell이 만들었으며, UI Widget의 느낌이나 개발자 경험이 기존에 존재하던 데스크탑용 애플리케이션 프레임워크(Java Swing, WinForms)를 흉내낸 것에 가까운 느낌이다. 당시에는 '웹에서도 데스크탑 앱 같은 경험을 구현해보자' 가 목표였기 때문이었던 것으로 짐작된다. 다만 지금의 Next.js와 같은 서버사이드 프레임워크가 아니라 클라이언트에서 로드되어 사용되는 런타임 라이브러리들의 집합이었다.

Dojo의 로더는 기본적으로 CJS를 흉내내기 위해 synchronous XHR(async off)로 런타임에 모듈을 로딩했다.

## 4. 서버사이드 모듈 시스템

그런데 뜬금없이 서버사이드 모듈 시스템이 먼저 CommonJS로 천하통일을 이루게 된다. 도대체 왜? 원래 문제는 클라이언트 모듈 시스템 아니었나?

그것은 2009년, 갑자기 Ryan Dahl의 Node.js가 등장하며 그들의 모듈 시스템으로 CJS를 채택했기 때문이다. Node.js는 서버사이드 자바스크립트에 대한 니즈가 너무 강력했다기보다는, Ryan Dahl의 기존 서버들의 concurrent connection의 한계에 대한 문제의식과 V8의 등장이 맞물려 탄생한 프로젝트가 개발자들의 니즈(isomorphic JS)와도 결합된 결과라고 볼 수 있다.

- 2009년 Kevin Dangoor의 [`What server side JavaScript Needs`](https://www.blueskyonmars.com/2009/01/29/what-server-side-javascript-needs/)

- [Naver D2 - JavaScript 표준을 위한 움직임: CommonJS와 AMD](https://d2.naver.com/helloworld/12864)

CJS를 브라우저 모듈시스템에서도 사용하기 위한 움직임들도 있었다. [curl.js](http://github.com/unscriptable/curl), [SproutCore](http://sproutcore.com), [PINF](http://github.com/pinf/loader-js)가 그것이다.

이들 curl.js와 PINF는 사실상 런타임에 비동기적으로 모듈을 로딩하는 방식으로 구현되었다.

## 5. 분리된 클라이언트 모듈 시스템 (AMD, Require.js)

CJS는 동기적으로 모듈을 로드하는 시스템이라서 브라우저에서는 채택이 어려웠다. 하지만 이미 대기업에는 클라이언트에서 사용되는 비동기 모듈 로드 시스템이 구현되어 있었고, 이런 구현들을 표준화 해 브라우저 측에서도 사용할 수 있는 표준 모듈 시스템을 구현하고자 하는 움직임이 일어났다. 그것이 AMD(Asyncronous Module Definition)이다.

Require.js는 AMD의 대표 구현체이다. [curl.js](https://github.com/cujojs/curl)와 경쟁구도가 있었으나 James Burke의 Require.js의 커뮤니티가 더 빠르게 성장하며 AMD를 대표하게 되었다.

AMD와 Require.js의 등장을 조금 자세히 설명하기 위해서는 위에서 등장한 Dojo toolkit과 James Burke의 관계에 대해 설명할 필요가 있다.

- [Require.js History](https://requirejs.org/docs/history.html)

## 6. 모듈 번들러(Browserify, Webpack)의 등장

AMD는 2011~2013년 즈음 득세하는 듯 했지만 Browserify, Webpack처럼 'CJS를 번들링해서 클라이언트에 던져주면, 클라이언트에서는 모듈 로딩이 필요없다' 라는 해결책이 나오면서 힘을 잃었다.

Browserify

Webpack은 CJS 뿐 아니라 AMD 코드도 지원했고, JS뿐 아니라 CSS, 이미지, 폰트 등 정적 에셋까지 모두 모듈로 묶었다(이게 뭔소릴까?).
