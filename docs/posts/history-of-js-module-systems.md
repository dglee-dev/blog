## 들어가며

자바스크립트는 복잡하다. 변화에 무한대로 열려있는 웹이라는 환경에서 실행되어야하고, 누군가 한 두명이 만들어서 배포하는게 아니라 마치 민주주의에서 법을 정하듯 위원회에서 언어의 규칙을 결정한다. 동적 언어로 설계되어 정적 분석이 어렵다는 등의 초기 설계에서의 태생적 한계도 존재한다.

자바스크립트는 1995년 탄생 이후 많은 변화를 겪어왔다. 비동기 요청의 탄생과 발전, V8, ES6, CommonJS, npm, TypeScript까지.. 여러 방면으로 확장하고 합의되며 그 영향력을 키워가고 있다. 그러므로 자바스크립트의 이 모든 변화를 하나의 주제만을 가지고 이해하기는 어렵다.

하지만 이 모든 변화들 중 가장 실용적이고 핵심적인 변화를 꼽자면 바로 **자바스크립트의 모듈시스템**이다. 본디의 가벼운 스크립트 언어에서부터 '진지한 프로그래밍 언어' 로 발전하는 과정의 중심에는 모듈시스템의 등장과 발전의 역사가 존재하며, 2026년 현재에도 이 모듈시스템을 둘러싼 발전은 진행형이다. 매일의 코드베이스에서도 우리는 아주 쉽게 이 발전의 흔적과 현재 진행중인 고민들을 확인할 수 있다.

개인적으로도 해당 주제에 닿고 본격적인 정리를 해야할 필요성을 느끼게 된 이유 또한 '매일 마주하는데 완벽히 이해하지 못하는' 상태를 아주 오랫동안 겪어왔기 때문이다. CJS와 ESM의 문법이 무엇이고 어떻게 사용되는지를 100번 찾아도 101번째 다시 찾게 되는 이유는 둘의 본질적인 차이와 그 맥락에 대한 몰이해 때문이라고 판단했다. tsconfig의 esModuleInterop 옵션을 설정할때도 이 설정이 왜 필요한 것인지 완전히 이해하지 못한 채 그저 필요하니까 켜고 끄는 일을 반복하며 죄책감을 느껴왔다.

이 죄책감을 해결하기 위한 방편으로, 단순히 Interoperability 문제를 해결할 수 있게 되는 것만이 아니라 Interoperability 문제가 왜 생겨났는지, 어떻게 해결할 수 있는 것인지, 옵션을 끄고 켤 때 내부적으로 어떤 일이 일어나는지 알기 위해 모듈시스템의 역사와 설계의 변천, 현재의 문제와 해결방법들까지 알아보려고 한다.

## 구성

해당 글은 3편의 시리즈로 작성될 예정이다.

1. 먼저 **모듈시스템의 발전**을 중심으로 자바스크립트의 역사에 대해 알아보고,
2. 현행으로 사용되고 있는 모듈 시스템인 **CJS와 ESM의 사용 방법과 설계의 특징**에 대해 알아본다.
3. 마지막으로 공존하는 현행 모듈 시스템 간의 **Interoperability**라는 현재 진행중인 문제를 TypeScript 나 번들러에서 어떻게 다뤄내고 있는지, 우리는 이 상황에서 어떻게 모듈을 작성하고 공개하며 조직해야 하는지에 대해 알아본다.

## 목표

3편까지 작성을 마쳤을 때, 쓰는 사람으로서는 정확한 어휘로 모듈시스템을 설명할 수 있는 사람이 되기를 바란다. 또한 모듈 시스템의 발전을 둘러싼 논의들을 따라가며 기술이 어떤 맥락과 동기를 기반하여 앞으로 나아가는지에 대해 이해하는, 기술안에 있는 사람들의 흔적을 알고 또한 자신의 결정에 그러한 기준들을 활용할 수 있는 사람이 되기를 바란다.

읽는 이는 왜 되거나, 왜 안되는지에 대한 이유를 알 수 있기를 바란다. require와 import는 왜 같이 사용할 수 없는지, package.json의 "type": "module" 이 어떤 상황에서 필요하며 정확히 무엇을 바꾸는지, esModuleInterop이 왜 필요한 옵션인지, 등등. 또한 파편화 된 지식들을 하나로 연결할 수 있는 접착제 역할을 할 수 있는 글이 되기를 바란다. CJS, ESM, 번들러, 트랜스파일러 등을 하나의 맥락 안에서 연결짓는다면 이들의 역할과 한계를 보다 확실하게 인식하여 현실세계의 프론트엔드 코드 인프라를 더 자신감있게 다룰 수 있게 되지 않을까.

# **Series 1: 자바스크립트 모듈 시스템의 역사**

## 1. AJAX로 인한 웹 애플리케이션의 등장

자바스크립트로 작성된 프로젝트들의 코드베이스의 규모가 커지게 된 것은 AJAX의 등장 덕분이었다. AJAX는 비동기적으로 데이터를 페칭하고, 가져온 데이터를 바탕으로 웹페이지를 refresh없이 업데이트하는 메커니즘을 의미하는 단어인데, 사실 2000년경 이미 Microsoft의 Outlook Web Access에서 이러한 비동기적 데이터 페칭과 부분적 화면 업데이트를 이용해 웹에서 데스크탑 앱과 같은 UX를 구현하였지만 AJAX라는 단어의 정립은 2005년에 Jesse James Garrett에 의해 이뤄졌으며 대중적으로 AJAX를 이용한 비동기적 화면 업데이트 방식의 웹 애플리케이션의 등장을 알린 것은 2004년에 등장한 Gmail이 대표적이다.

이렇게 AJAX를 이용해 하나의 웹페이지를 마치 애플리케이션처럼 개발할 수 있게 되면서 UI 상태관리, DOM 조작, 비동기 흐름 제어, 클라이언트 사이드 라우팅 등을 위한 코드가 자바스크립트 코드베이스에 추가되기 시작했다. 우리가 알고 있는 '프론트엔드 개발' 시대가 시작된 것이다. 이렇게 클라이언트 측 자바스크립트의 코드량이 증가하게 되면서, 자연스럽게 **모듈 시스템에 대한 필요성**이 대두되게 되었다.

예를 들어, 지메일과 같은 메일 클라이언트 웹앱에서 사용되는 복잡한 로직들을 각각의 모듈로 만든다고 하면 아래와 같이 분리될 수 있을 것이다.

```
core/
api/
mail/
draft/
search/
router/
ui/
keyboard/
cache/
notification/
```

여기에서 의존성을 고려해서 스크립트를 로드하려면 어떤 모듈이 어디에 의존하는지를 기록하는 문서를 작성하게 될 것이다. 그리고 해당 문서에 의거해서 스크립트 순서를 개발자가 수동으로 관리해야한다. 이 부분에서 개발자라면 '작성된 의존성대로 스크립트 순서를 자동으로 계산해주는 알고리즘을 짤 수 있지 않을까?' 라는 생각이 들기 마련이다. 이것이 당시의 `Loader` 의 역할이다. 여기에 중복 로딩을 방지하고, race condition, 로딩 실패처리, 캐시, 실행 순서, circular dependency등에 대한 처리를 추가하다보면 사실상 작은 런타임 시스템이 필요해지는 것이다.

그렇게 대규모 웹 애플리케이션 작성을 위한 초기 클라이언트 모듈 시스템들이 생겨나게 되었다.

## 2. 초기 클라이언트 모듈 시스템

### Google Closure Library, Yahoo User Interface Library

웹페이지의 애플리케이션화로 인해 코드가 복잡해지고 이에 따라 모듈 로더에 대한 필요성이 생겨나면서, 대기업들은 그들의 자체적인 기술력을 이용해 클라이언트 모듈 시스템을 제작해냈다. Gmail을 개발한 구글은 이미 2004년경 현대의 Webpack과 같은 역할을 하는 번들링/컴파일 툴인 `Closure Library`를 만들어 사용하고 있었으며, 야후는 [`YUI`](https://clarle.github.io/yui3/) 라는 이름의 라이브러리에 모듈 로더를 얹어내었다.

### Dojo Toolkit

Dojo는 2004년에 등장한 프레임워크로서, Dijit이라는 이름의 UI 위젯 라이브러리와 Java/C# 친화적인 클래스 기반 OOP 패러다임(dojo.declare), 대규모 자바스크립트 코드베이스를 관리하기 위한 모듈 시스템(sync XHR - AMD), 그리고 i18n과 크로스 브라우징까지 하나로 묶어 제공하는 '엔터프라이즈급' 웹 애플리케이션 제작용 툴킷이었다. Alex Russell이 만들었으며, UI Widget의 느낌이나 개발자 경험이 기존에 존재하던 데스크탑용 애플리케이션 프레임워크(Java Swing, WinForms)를 흉내낸 것에 가까운 느낌이다. 당시에는 '웹에서도 데스크탑 앱 같은 경험을 구현해보자' 가 목표였기 때문이었던 것으로 짐작된다. 다만 지금의 Next.js와 같은 서버사이드 프레임워크가 아니라 클라이언트에서 로드되어 사용되는 런타임 라이브러리들의 집합이었다.

Dojo의 로더는 기본적으로 CJS를 흉내내기 위해 synchronous XHR(async off)로 런타임에 모듈을 로딩했다.

## 3. 서버사이드 모듈 시스템(CJS)과 Node.js

그런데 뜬금없이 서버사이드 모듈 시스템이 먼저 CJS로 천하통일을 이루게 된다. 도대체 왜? 원래 문제는 클라이언트 모듈 시스템 아니었나?

그것은 2009년, 갑자기 Ryan Dahl의 Node.js가 등장하며 그들의 모듈 시스템으로 CJS를 채택했기 때문이다. Node.js는 서버사이드 자바스크립트에 대한 니즈가 너무 강력했다기보다는, Ryan Dahl의 기존 서버들의 concurrent connection의 한계에 대한 문제의식과 V8의 등장이 맞물려 탄생한 프로젝트가 개발자들의 니즈(isomorphic JS)와도 결합된 결과라고 볼 수 있다.

- 2009년 Kevin Dangoor의 [`What server side JavaScript Needs`](https://www.blueskyonmars.com/2009/01/29/what-server-side-javascript-needs/)

- [Naver D2 - JavaScript 표준을 위한 움직임: CommonJS와 AMD](https://d2.naver.com/helloworld/12864)

CJS를 브라우저 모듈시스템에서도 사용하기 위한 움직임들도 있었다. [curl.js](http://github.com/unscriptable/curl), [SproutCore](http://sproutcore.com), [PINF](http://github.com/pinf/loader-js)가 그것이다.

이들 curl.js와 PINF는 사실상 런타임에 비동기적으로 모듈을 로딩하는 방식으로 구현되었다.

## 4. 분리된 클라이언트 모듈 시스템 (AMD, Require.js)

CJS는 동기적으로 모듈을 로드하는 시스템이라서 브라우저에서는 채택이 어려웠다. 하지만 이미 대기업에는 클라이언트에서 사용되는 비동기 모듈 로드 시스템이 구현되어 있었고, 이런 구현들을 표준화 해 브라우저 측에서도 사용할 수 있는 표준 모듈 시스템을 구현하고자 하는 움직임이 일어났다. 그것이 AMD(Asyncronous Module Definition)이다.

Require.js는 AMD의 대표 구현체이다. [curl.js](https://github.com/cujojs/curl)와 경쟁구도가 있었으나 James Burke의 Require.js의 커뮤니티가 더 빠르게 성장하며 AMD를 대표하게 되었다.

AMD와 Require.js의 등장을 조금 자세히 설명하기 위해서는 위에서 등장한 Dojo toolkit과 James Burke의 관계에 대해 설명할 필요가 있다.

- [Require.js History](https://requirejs.org/docs/history.html)

## 5. 모듈 번들러(Browserify, Webpack)의 등장

AMD는 2011~2013년 즈음 득세하는 듯 했지만 Browserify, Webpack처럼 'CJS를 번들링해서 클라이언트에 던져주면, 클라이언트에서는 모듈 로딩이 필요없다' 라는 해결책이 나오면서 힘을 잃었다.

Browserify

Webpack은 CJS 뿐 아니라 AMD 코드도 지원했고, JS뿐 아니라 CSS, 이미지, 폰트 등 정적 에셋까지 모두 모듈로 묶었다(이게 뭔소릴까?).
