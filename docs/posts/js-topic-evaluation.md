## 1. 주제 후보

### 언어 자체의 표현력

- **모듈시스템** — CJS, AMD, UMD, ESM, `esModuleInterop` 등
- **비동기 모델** — 콜백 → Promise → async/await → top-level await
- **타입 시스템** — JSDoc → Flow vs TypeScript → TS 표준화 → 네이티브 타입 스트리핑
- **클래스/객체지향 문법** — 프로토타입 기반 상속 → `class` 문법 → "문법 설탕" 논쟁
- **에러 핸들링** — `(err, data)` 콜백 패턴 → try/catch·rejection → `Result` 타입 제안
- **불변성/함수형 패러다임** — lodash/Ramda → `Object.freeze` → Records & Tuples 제안

### 실행 환경 / 런타임

- **엔진 성능 경쟁** — SpiderMonkey → V8 JIT 혁신 → Node.js 탄생의 전제조건
- **런타임 다양화** — Node.js → Deno → Bun
- **브라우저 API 확장** — XHR → Fetch → Service Worker → WebAssembly

### 생태계 / 배포

- **패키지 매니저** — npm → yarn → pnpm, 의존성 지옥과 해법
- **번들러/빌드 도구** — Browserify → webpack → Rollup → esbuild/Vite
- **트랜스파일러 역할 변화** — Babel의 탄생 이유와 현재의 존재 의의

### UI / 애플리케이션 레이어

- **DOM 조작 방법론** — jQuery → Virtual DOM → Signals 기반 반응성
- **컴포넌트 모델과 프레임워크 전쟁** — Backbone → Angular → React/Vue → 메타프레임워크
- **스타일링 방법론** — 인라인 스타일 → CSS 클래스 → CSS-in-JS → 유틸리티 퍼스트
- **상태관리** — 전역 변수 → Flux/Redux → Context API → 서버 상태 분리

### 테스팅 / 품질

- **테스트 문화** — Mocha/Jasmine → Jest → Vitest
- **린팅/포매팅** — JSLint → ESLint → Prettier → Biome

### 메타적 주제

- **자바스크립트 피로감(JS Fatigue)** — 2016년경 밈이 된 현상 자체
- **표준화 정치** — TC39 stage 프로세스, 벤더 간 힘겨루기

## 2. 평가 기준

### 실용적 가치 축

**① 일상적 마주침 빈도**
개발자가 실제 코드베이스에서 이 주제의 흔적(설정, 문법, 에러메시지)을 얼마나 자주 마주치는가.

**② 오해로 인한 실질적 비용**
이 주제를 잘못 이해했을 때 실제로 버그, 배포 실패, 디버깅 시간 낭비로 이어지는 정도.

**③ 설정/옵션의 불투명성**
"왜 켜야 하는지 모른 채 쓰는" 블랙박스 설정이 얼마나 많이 딸려 있는가.

### 언어/생태계 내 위치 축

**④ 구조적 중심성**
이 주제가 언어의 다른 부분들과 얼마나 얽혀 있는가 — 이해하면 다른 주제도 같이 풀리는 "허브" 역할을 하는지.

**⑤ 진행 상태(현재진행형일수록 높음)**
이 논쟁/전환이 역사적으로 끝났는가, 아니면 여전히 현재진행형이라 서사적 긴장이 남아있는가.

### 서사적 가치 축

**⑥ 극적 갈등의 존재**
그 역사 안에 뚜렷한 대립 구도, 실패, 반전이 있는가.

**⑦ 반직관성 / 통찰의 밀도**
표면적으로 알던 사실 뒤에 "몰랐는데" 하는 지점이 얼마나 있는가.

**⑧ 은유/서사 프레임과의 정합성**
"도시(지층, 인프라)" 비유와 그 주제가 얼마나 자연스럽게 맞아떨어지는가.

### 실행 가능성 축

**⑨ 자료의 확보 가능성**
스펙 문서, 커밋 로그, 당시 블로그·메일링 리스트 등 1차 자료가 충분히 남아있는가.

**⑩ 스코프 통제 가능성**
주제가 무한히 확장될 위험이 있는가, 아니면 자연스러운 시작과 끝이 있는가.

### 짧은 해석

- **모듈시스템**이 압도적 1위예요 — 빈도, 불투명성, 중심성, 은유 정합성 네 축 모두에서 최상위권.
- **번들러/빌드 도구**와 **비동기 모델**이 근소하게 2, 3위를 다투는데, 성격이 달라요. 번들러는 모듈시스템과 사실상 절반쯤 겹치는 인접 주제(같이 다뤄도 자연스러움)고, 비동기 모델은 실용적 비용(②)은 가장 높지만 은유 정합성(⑧)이 낮아서 지금 잡은 "도시" 프레임 안에서는 억지스러울 수 있어요.
- **표준화 정치(TC39)**는 중심성·진행형·자료성에서 높은 점수를 받지만 일상 빈도(①)가 낮아서, 본론보다는 배경 설명이나 도입부 장치로 쓰기 좋은 위치예요.
- 하위권(린팅/포매팅, 테스트 문화, 불변성)은 대체로 "스코프는 통제하기 쉽지만 서사적 긴장이 약한" 주제들이라, 메인 주제보다는 짧은 곁가지 언급 정도로 어울려요.
